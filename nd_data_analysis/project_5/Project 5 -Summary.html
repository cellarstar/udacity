ANSWER:¶

The goal of the project is to identify POIs (person of interest) of the Enron fraud case by analysing a data set consisting of email data of numerous Enron employees as well as financial information about the employees. Some of the POIs have already been identified and those have been labelled accordingly in the dataset already. For our investigations, this is relevant as it allows us to use supervised learning algorithms for identifying further POIs in the dataset. The available email and financial data will be used to training and test different learning algorithms as well as different configurations of the complete test cases (algorithm and data). Two considerable outliers have been removed from the dataset before further performing analysis. Those have been identified doing a qualitative analysis of the data, namely taking a look at the data in an appropriate application (eg. Excel) and searching manually for data which seems to be inappropriate for using it as a base for further investigations. Once identified, that data can be removed using Python code.

The above list includes both, original features of the data set, as well as newly created features. In general, newly created features based on financial data can be identified by the suffixes _log and _squared. New features created based on the messaging data are total_messages, total_poi_related_messages, and total_poi_message_ratio. New financial features are based upon salary, expenses and bonus only (as those are chosen for the final data set used to train the classifier). New messaging features are constant, means not depending on the selection of messaging features. Obviously, new features built on financial features, very likely show the same number of NaN values.

One example of a newly created feature is total_poi_related_messages. This feature shows the total number of touchpoints a person has with a POI. Means it combines the number of messages sent from a POI to that person, the number of messages a that person sent to a POI, and the number of messages where both, that person and the POI, have received the same message. The idea was to find a number describing a kind of aggregated view of all availalbe message related data for a person in relation to POIs.

The final classifier will use the original features only, nevertheless I will quickly show the differences in performance of the classifier using original features only and using original features plus newly created features. However, I will not elaborate on the full spectrum of features, but focus only on 6 original features (salary, bonus, expenses, from_poi_to_this_person, from_this_person_to_poi, shared_receipt_with_poi). Those 6 original features represent my final data set for this project.

ANSWER:¶

From the email data the employees email address has been removed from the data as it does not make sense to use it for any investigations, basically it is a unique identifier for a person, but that's it.

My approach on the remaining features was changing over time. In a first attempt, the preparation of features was done using a PCA algorithm. No manual pre-exclusion of features was done. However, the programm allowed to define the number of features used for the learning process. Nevertheless, the automatic composition of the features based on a PCA algorithm was not sucessfully (as the result I was able to achieve did not satisfy the specification limits) and even using a univariate feature selection algorithm did not significantly improve the results. Finally, I had to change from an automatic, completely machine driven approach to a semi-manual one over time. I ended up using features which I have manually chosen based on their scores (SelectKBest) and importances (importances_ scores) in combination with human intuition and tested them using different classifiers. Finally was able to reach the requested results. More information about the different approaches follows below.

First attempt

From the financial data set non of the features have been removed manually, the approach was to apply a PCA (principal component analysis) algorithm to optimize the features for further downstream analysis with supervised learning classifiers.

Some of the financial data has been used as a base to create new features. Basically, the features process included a log as well as a square transformation of the data aiming to better visualize differences in the different values or identifying trend which are hidden in the raw data set. Further, some new KPIs for the email data have been created to even better understand the behaviour of POIs, especially their involvement in different email conversations. Eg. one feature provides information about all POI related messages of a specific person, an other one provides the relationship between the number all those POI related messages and all messages. Those new features target to further support analysis by providing a new views from different angles. All newly created features have been included into the features selection process performed by the PCA algorithm.

For performing the PCA, a list numbers of target parameters has been provided. The program was checking all the different possibilities and for each of the different options the different classifiers with different configurations were applied. The best performing configuration was picked at the end. Scores for the different configurations are provided as output in the program.

However, the approach of automatically selecting features has not been successfully. Even though I tried lots of different classifiers with various configurations, the classifiers have not been satisfying from a precision and recall perspective.

Second attempt

I reworked what I have done from an automatism perspective and created a program which was allowing for a more semi-automatical selection of the algorithm, its configuration, the number of features used and which features to be used. Following a manual selection of the features to be used for the classification, I needed some support to make my descision. Therefore I implemented a section to print the feature score based on a univariate feature selection algorithm, namely SelectKBest. Further, I included one section to print the importance of features for a specific algorithm. Based on that information, I was able to sort our useless features and concentrate on the important ones. By doing that, I was able to clearly increase the performance of the algorithms. The features I finally used have been already discussed as part of the first question.

A description of the steps for features selection and classifier tuning:

0) Automatic selection of best features based on SelectKBest algorithm. Result: Take a look at features_scores.csv to see how the different features have been rated by SelectKBest algorithm in combination with an ANOVA F-value function, as I found this to be a good function to use in case of a classification problem. However, when using the features proposed by SelectKBest, I did not get satisfying result. Therfore, I trashed this approach again and had to proceed as described in the next steps.

1) Based on some basic tests (using the results of feature selection algorithm SelectKBest and the importance performance indicator as a guideline) combined with human intuition (eg. if it comes to fraud detection of employees, _bonus_ seems to be an interesting feature even before you prove the assumption by performing different tests). The goal of that step was to find the features which are going to be used for further tunig of the algorithm as well as the top classifiers. Result: Features - bonus, expenses, total_payments, director_fees, from_poi_to_this_person; Classifiers - AdaBoostClassifier, KNeighborsClassifier

2) Use the best performing algorithms (AdaBoostClassifier, KNeighborsClassifier) and find the best performing features (base for this are the result of step 1 - bonus, expenses, total_payments, director_fees, from_poi_to_this_person) and related options like considering newly created features and using PCA.

QUESTION 5:¶

What is validation, and whatâs a classic mistake you can make if you do it wrong? How did you validate your analysis?

ANSWER:¶

Validation means to test your algorithm after you configured and trained it. A good approach for the overall process is:

1) split your data into train and test data
2) do a PCA based on your training features (fit, transform)
3) pick, configure and train your classifier (use the PCA transformed train data)
4) transform your test data (for validation) using PCA
5) predict your results based on the test data
6) get relevant information about the KPIs (accuracy,..)

To get better validation results, specific approaches have been developed to support this process. An example for such an approach are cross validation algorithms. When using cross validation, the data set gets split into training and test sets to simulate how the model will behave once it starts predicting on new data.

QUESTION 6:¶

Give at least 2 evaluation metrics and your average performance for each of them. Explain an interpretation of your metrics that says something human-understandable about your algorithm’s performance.

Recall: Out of all the items that are truly positive, how many are correctly classified as positive. How many positive items were recalled from the data set. In other words, how good is the algorithm in recognizing items which are actually belonging to a specific class. For example: How strong is the algorithm in recognizing that "BOWEN JR RAYMOND M" is a POI and put him into the POI class.

Prevision: Out of all the items labeled as positive how many truly belong to the positive class. In other words, how good is the algorithm in terms of correct class assignment. For example: What is the proportion of POI's in the POI class considering all items which have been identified and assigned to the POI class by the algorithm.

Metrics of the best performing algorithm:

  • Precision: 0.35871
  • Recall: 0.33450

A common mistake which can be made in the validation phase is using the same data which was earlier used for training the classifier, for validating it. Obviously, that is not a valid validation case as training data and validation data have to be different.

Validation used in the project¶

StratifiedShuffleSplit in combination with GridSearchCV have been used in poi.py to find out the best configuration for an algorithm. The reason for using StratifiedShuffleSplit was the relatively low number of training and testing data.

Also tester.py used StratifiedShuffleSplit. I have changed the source code of tester.py a bit to get metrics of tests returned to where the function test_classifier is called, to be afterwards able to store and compare the results.

StratifiedShuffleSplit algorithm shuffles the data around before it gets split. The number of such shuffle/split operations can be provided as a parameter to the algorithm. Further test split between test and training data can be defined as well as a pseudo-random number generator state used for random sampling.
